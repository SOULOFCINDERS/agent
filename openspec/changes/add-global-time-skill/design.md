## Context

当前项目的 Agent 采用 `Planner -> Executor -> Tool Registry -> Tool` 的执行链路，新增能力通常通过实现一个新工具、在 Planner 中增加触发规则、在 CLI 中注册来完成。查询全球时间本质上是一个确定性能力，适合优先使用 Go 标准库 `time` 直接完成，而不是依赖外部 HTTP API。

主要约束如下：
- 现有工具接口为同步调用，需要返回简洁文本结果，便于链式传递给后续步骤
- Planner 当前是规则驱动，需要通过前缀或关键字识别时间查询请求
- 工具应尽量避免新增外部依赖，保证测试稳定性与离线可运行性

## Goals / Non-Goals

**Goals:**
- 新增一个可注册的全球时间查询工具
- 支持城市名与 IANA 时区两种输入形式
- 提供稳定、可读的文本输出，适合继续链式处理
- 使该能力可通过显式命令和简单自然语言被 Planner 识别

**Non-Goals:**
- 不实现复杂的日程安排或会议时间推荐
- 不实现完整的时区知识库服务
- 不处理历史时间、未来时间预测或 DST 规则解释型问答
- 不引入新的外部天气/时间 API 依赖

## Decisions

### 1. 使用 Go 标准库 `time.LoadLocation`
使用标准库时区数据库加载 IANA 时区，例如 `Asia/Shanghai`、`Europe/London`。这样无需网络请求，结果稳定，且与 Go 运行环境原生兼容。

**Alternatives considered:**
- 外部时间 API：实现简单，但引入网络不稳定与测试波动
- 固定偏移量计算：难以正确处理夏令时，不可靠

### 2. 增加少量城市别名映射
为了支持 `Tokyo`、`Beijing`、`New York` 这类用户常见输入，在工具内部维护一份轻量别名映射，将城市名归一化到标准 IANA 时区。

**Alternatives considered:**
- 仅支持 IANA 时区：实现更简单，但用户体验较差
- 引入完整地理数据库：过重，超出当前范围

### 3. 输出统一为单行摘要文本
返回格式类似 `Tokyo (Asia/Tokyo): 2026-04-06 21:30:00 JST`。这既方便人类阅读，也方便下一步 `summarize` 或 `echo` 等工具消费。

**Alternatives considered:**
- 返回复杂 JSON：机器可读性更强，但当前 CLI 文本交互场景不需要
- 返回多行格式：可读性稍强，但不利于链式处理

### 4. Planner 增加前缀与自然语言双触发
Planner 同时支持：
- 显式前缀：`time: Tokyo`
- 自然语言：`东京现在几点`、`time in London`

这样可以兼顾可控性与易用性。

## Risks / Trade-offs

- [城市别名覆盖不足] → 先覆盖高频城市和主要时区，无法识别时回退为明确报错，后续按需要补充映射
- [运行环境缺少完整时区数据] → 优先使用 Go 自带能力；若个别环境异常，在 README 中给出已知限制说明
- [自然语言规则误判] → 优先保守匹配，只为明确“时间/几点/time”语义触发，避免影响其他工具
