# 在本项目实现一个可迭代 Agent：实施计划

## 目标
在现有项目代码库中落地一个“可交互、可迭代”的 Agent（先最小可用，再逐步增强），并在实现过程中通过小步提交式的迭代，让你能边写边学，最终形成可运行的 CLI/服务入口、可扩展的 Agent 内核、可测试的工具体系与示例。

> 说明：当前工作区看起来只有 `.git/` 元数据，尚未检出工作树文件。计划将先解决“代码不在工作区”这一阻塞，再进入 Agent 设计与实现。

## 假设与边界（先按最合理默认推进，后续可调整）
- **语言与工程形态**：优先按 Go 项目落地（路径在 `go/src` 下且仓库名为 `agent`）。若检出后发现是 Node/Python/Rust，将按实际工程调整实现方式。
- **Agent 的定义**：先实现一个“具备工具调用能力的任务执行 Agent”（Planner/Executor 或 ReAct 风格均可），而不是单纯聊天机器人。
- **不引入真实密钥**：默认不直接接入在线 LLM（避免密钥/费用/稳定性问题），先做可插拔 `LLMClient` 接口 + 本地 Mock/规则引擎；之后再可选接入 OpenAI/火山/豆包等。

## 里程碑与交付物
1. **工作区可构建**：代码完整检出、依赖可安装、基本构建/测试命令可跑通。
2. **最小垂直切片（MVP Agent）**：
   - 一个可运行入口（CLI：`agent run "<task>"` 或类似命令）
   - 一个 Agent 核心接口（`Plan`/`Step`/`ToolCall`）
   - 工具注册表与 2–3 个内置工具（如 `echo`、`calc`、`read_file`）
   - 一条可演示的端到端流程（输入任务 → 选择工具 → 执行 → 汇总输出）
3. **可测试与可扩展**：
   - 单测覆盖核心 Planner/Executor 与工具调用
   - 配置体系（YAML/JSON/ENV）与日志
   - 文档与示例（README/示例任务）
4. **增强能力（按需迭代）**：记忆、反思/重试、并发工具、权限沙箱、检索、工作流编排等。

## 实施步骤（按顺序执行）

### 第 0 步：恢复/检出工作树（阻塞处理）
- 检查仓库是否处于 sparse-checkout / 未检出 / 子模块未初始化等状态。
- 执行一次完整检出（如 `git checkout main`、`git reset --hard origin/main`、必要时 `git submodule update --init --recursive`）。
- 验证工作区出现项目文件（如 `go.mod` / `package.json` / `README` / `cmd/` 等）。

**验收**：根目录不再只有 `.git/`；可以看到项目源码与构建文件。

### 第 1 步：快速摸底现有工程与约束
在不改动代码的前提下完成一次结构与规范调查：
- 工程语言与构建工具：Go / Node / Python / 混合？
- 入口位置：`cmd/`、`main.go`、`src/`、`apps/` 等
- 配置/日志/依赖管理：现有库与风格（例如 zap/logrus、cobra/urfave 等）
- 是否已有 “agent/runner/tool” 等概念雏形（复用而非另起炉灶）

**产出**：一份“工程现状摘要 + 我们要插入的模块边界”。

### 第 2 步：定义最小 Agent 能力（需求切片）
把“agent”落到可实现的最小集合：
- 输入：一段任务文本（后续可扩展为结构化任务）
- 输出：执行结果（文本/JSON）
- 中间态：`Plan`（若干步骤），步骤可为 `ToolCall` 或 `FinalAnswer`
- 工具：先做 2–3 个纯本地、确定性的工具，便于测试与迭代

**验收**：有一条清晰的端到端用例（示例任务与预期输出）。

### 第 3 步：设计与落地核心抽象（先简单、可替换）
按“可插拔、可测试”设计最小架构（会贴合现有代码风格）：
- `Agent`：对外 `Run(ctx, input) (output, error)`
- `Planner`：把输入转成步骤（最初可用规则/模板实现）
- `Executor`：执行步骤（工具调用、错误处理、重试策略）
- `Tool`：`Name/Description/Schema/Execute`
- `ToolRegistry`：注册/查找/权限控制
- `LLMClient`（可选）：接口先定义，默认用 Mock/本地策略

**验收**：接口清晰，核心逻辑可以在不依赖外部服务的情况下跑通并单测。

### 第 4 步：实现 MVP（CLI + 内核 + 内置工具）
按垂直切片推进，确保“每次改动都能跑”：
- 增加 CLI 命令（例如 `agent run`），支持读取任务、输出结果、可选 `--trace` 打印步骤
- 实现 `Planner`（第一版：规则驱动；第二版：接入可选 LLM）
- 实现 `Executor`（执行 ToolCall、捕获错误、汇总输出）
- 内置工具建议：
  - `echo(text)`：回显
  - `calc(expression)`：简单四则运算（避免引入重依赖）
  - `read_file(path)`：受控读取（路径白名单/最大大小限制）

**验收**：`agent run "读取某文件前 20 行并总结"`（或类似任务）可成功完成。

### 第 5 步：测试与质量保障
补齐可持续迭代的护栏：
- 单元测试：Planner/Executor/Tool
- 端到端测试：CLI 运行一条固定脚本
- 静态检查：`go test ./...`、`go vet`、`golangci-lint`（若项目已有）
- 错误与日志规范：避免输出敏感信息

**验收**：CI/本地测试全绿（以项目既有命令为准）。

### 第 6 步：按你的学习节奏逐步增强（迭代清单）
根据你想要的“agent 能力”选择迭代方向，每次只加一个小能力并验收：
- 记忆：会话级 / 任务级（本地文件或内存）
- 反思与重试：失败时自动改写计划重试
- 多工具编排：并行执行与依赖图
- 权限与沙箱：工具访问控制、路径/网络限制
- 检索：本地代码检索/RAG（如需要）
- 可观测性：trace、step 记录、可视化

## 执行方式（互动与指导）
- 我会把实现拆成“小步可运行”的增量：每一步都给你明确的运行方式与验证点。
- 你可以随时调整目标（例如想做“自动写代码的 agent”或“客服 agent”），我会在不推翻整体架构的前提下迭代。

## 风险与应对
- **仓库未检出导致无法实施**：先完成第 0 步；若远程不可访问，改为在当前目录初始化一个最小 Go 工程并继续。
- **外部 LLM 接入复杂**：先做接口与 Mock；需要时再加真实 Provider，并通过环境变量注入 Key。
- **工具安全**：文件读取/命令执行默认强限制，逐步放开并可配置。
